# Workstream scout — the system's first divergent agent

The operationalization of the charter reframe: now that the
[Mission is the filter and the Core Axioms are the screen](2026-06-27-charter-restructure.md),
an agent can *use* them to propose the right next initiatives. Every agent today
is convergent (reviews, gates, the gardener — they guard quality and say *no*);
this is the first **divergent** one — its job is to open possibility and say
*"what if."*

## Problem

The charter now declares that "every proposed workstream is measured against the
Mission" — but nothing *generates* those proposals. Choosing the project's next
big initiatives is today an unstructured, ad-hoc act that happens only when a
human happens to think about it. Three gaps, observably:

1. **No divergent engine.** The harness can plan, build, review, and remember a
   *given* initiative, but it has no machinery to ask "what should the next
   initiative even be?" — and certainly not one that generates *several*
   competing visions and pits them against each other.
2. **No outward sight.** Every persona reads only this repo's own docs. None can
   look at the competitive frontier, how peer agent-platforms / dev-tooling
   evolved, or draw analogies from other domains — the "meta, beyond the project"
   perspective that separates a real strategist from an incrementalist.
3. **No disciplined place for a foundational challenge.** When a compelling
   direction would require bending the Mission or breaking an axiom, today it is
   either silently ignored (too rigid) or would be silently acted on (too
   reckless). There is no channel that says "this idea is good enough that the
   *axiom itself* deserves a human re-decision."

## Requirements

- **R1.** A `scout` skill exists that, on demand, produces a ranked strategic
  proposal for the project's next initiatives and commits it as a durable doc.
  (Verify: invoke it; a `docs/horizons/` doc appears, gate GREEN.)
- **R2.** The proposal is generated by a **divergent panel**: ≥3 vision
  generators, each forced into a *distinct stance* (default set: moonshot,
  competitor-killer, first-principles-reframe, narrowest-wedge), so the visions
  are genuinely different rather than variations of one idea. (Verify: the
  proposal records each vision's stance; the visions are materially distinct.)
- **R3.** Each generator performs **outward research** (web) within its stance —
  competitive frontier, peer systems, cross-domain analogies — and cites what it
  found. (Verify: visions carry external citations, not only repo-internal
  reasoning.)
- **R4.** An **independent** evaluator (`vision-judge`, not the generator) scores
  every vision against a fixed rubric — Mission-alignment, axiom-fit,
  competitive edge, leverage, feasibility — and **routes** each to exactly one of:
  **Tier 1** (actionable initiative, within Mission + all axioms), **Tier 2** (a
  *foundational challenge* — compelling but requires evolving a charter
  commitment: a Mission clause, a Core Axiom, or a load-bearing
  design-philosophy / identity commitment), or **dropped** (with reason). (Verify:
  every vision has an independent
  score sheet and a tier with justification; no generator grades its own vision.)
- **R5.** The `horizon` proposal doc has two tiers: **Tier 1** ranked initiatives
  (each: the vision, why it wins, the rubric scores, and a suggested first
  wedge), and **Tier 2** flagged foundational challenges (each: the idea, which
  charter commitment it strains — a Mission clause, a Core Axiom, or a load-bearing
  design-philosophy / identity commitment — and why it might be worth the human
  re-deciding). Tier 2 is **rare and explicitly marked**. (Verify: read the doc;
  both tiers present with the required fields, Tier 2 names the strained
  axiom/clause.)
- **R6.** The scout **proposes, never enacts** — it writes the proposal doc and
  stops. It does not create product-specs, ExecPlans, tickets, or edit the
  charter. (Verify: a scout run's diff is exactly one new `docs/horizons/` doc +
  index registration — nothing else.)
- **R7.** The new `horizon` doc type is registered in the knowledge format's type
  table (self-host + seed) so it is a first-class, discoverable artifact with
  conformant frontmatter (`type: horizon`, `description`). (Verify: the type
  table lists it; `check.py` accepts the new doc.)

## Design

Three new components in the portable layer (`plugin/`) plus one new doc type. The
shape mirrors the repo's existing **skill-orchestrates-persona** pattern (`garden`
→ `doc-gardener`), extended to a fan-out: one orchestrating skill, two persona
roles (divergent generator, convergent judge), and the skill itself does the
context-bound synthesis.

### Component 1 — `scout` skill (`plugin/skills/scout/SKILL.md`)

The thin orchestrator, run by the main session (which holds full project
context). Procedure:
1. **Ground in the project.** Read `docs/CHARTER.md` (Mission + axioms),
   `nav.py roadmap` (what exists / what's in flight), `docs/logs.md` (what's been
   tried and retired — so visions don't re-propose dead ends).
2. **Fan out generators.** Dispatch the `workstream-scout` persona once per stance
   (default 4), each via the Task tool, passing the stance + the project grounding.
   (These are ordinary subagent dispatches — the same mechanism the review
   personas use; no Workflow-tool dependency.)
3. **Judge independently.** For each returned vision, dispatch a *fresh*
   `vision-judge` (one per vision, blind to siblings) to score + tier it. A
   generator never judges its own (or any) vision — independence is the point.
4. **Synthesize (own context).** The skill — not a subagent — does the
   comparative ranking within Tier 1 and assembles the two-tier `horizon` doc,
   because cross-vision comparison and final framing need the full project context
   the orchestrator holds (DESIGN.md: "construction that needs the repo's FULL
   context is a skill"; own-context-for-synthesis).
5. **Write + commit.** Write `docs/horizons/YYYY-MM-DD-<slug>.md`, register it in
   a `docs/horizons/index.md`, run the gate to GREEN, commit (scoped `git add`,
   never `-A`).

The skill is **Director/main-session-side by intent** — the scout sets direction,
which ticket-executing workers do not, so a worker never invokes it. This is a
documented intent, **not a mechanically-enforced exclusion**: `install_worker_methodology`
currently blanket-copies all skills/agents into worker runtimes, so the files do
land there (harmless — a worker only runs a skill its ticket names). A per-component
`scope: director` exclusion is tracked tech-debt, best landed with the in-flight
worker-vendoring redesign. Cadence is
**on-demand** (the human runs it when thinking about what's next); periodic /
completion-triggered invocation is a Non-goal here (follow-up).

### Component 2 — `workstream-scout` persona (`plugin/agents/workstream-scout.md`)

The divergent generator. Frontmatter `tools: Read, Grep, Glob, WebSearch,
WebFetch` (web research is the new capability; **no Edit/Write** — it returns its
vision, it does not author the doc). Primary grounding: `docs/CHARTER.md` (the
Mission is the altitude a vision must clear). Mandate, in the body:
- Take the **assigned stance** as a forcing function for divergence, and reason
  from **first principles** about what the project could become under it — not the
  next increment.
- **Research outward**: the competitive frontier, how peer agent-platforms /
  dev-tooling / orchestration systems evolved, and **analogies from unrelated
  domains**. Cite sources.
- Return a structured vision: the bet, the first-principles argument, external
  evidence, the competitive edge, the leverage, and an honest "what this would
  cost / what it strains" (including whether it appears to need an axiom to bend —
  a signal the judge will weigh).
- It does **not** self-censor against the axioms (that is the judge's job) —
  divergence is maximal at generation; the screen comes later.

### Component 3 — `vision-judge` persona (`plugin/agents/vision-judge.md`)

The convergent, independent evaluator. `tools: Read, Grep, Glob, WebFetch` (read
to verify a cited claim). Primary grounding: `docs/CHARTER.md` (Mission + Core
Axioms = the rubric); secondary input `docs/PRINCIPLES.md` (to predict what the
human would value). For one vision it returns a score sheet
(Mission-alignment / axiom-fit / competitive-edge / leverage / feasibility, each
scored + justified) and **routes**:
- **Tier 1** if it clears the Mission and violates no axiom.
- **Tier 2** if it is compelling *but* its core requires evolving the Mission or
  bending a specific axiom — the judge names which clause/axiom and why it could
  be worth a human re-decision. This is the keystone: the axiom-screen is a
  **router, not just a filter** — a brilliant axiom-breaking idea is neither
  killed nor enacted, it is escalated.
- **dropped** with a one-line reason (off-Mission, low leverage, already tried per
  logs, no real edge).

### New doc type — `horizon`

`type: horizon` registered in `docs/KNOWLEDGE_FORMAT.md` + its seed
`plugin/skills/harness-init/templates/knowledge-format.md` (type *values* are free
in KF v2.0, so this is additive — presence-only, no schema change). Lives under a
new `docs/horizons/` directory with an `index.md`. It is a **proposal artifact**,
not a spec or plan — it carries no `phase:` and never appears in the roadmap; it
is decision-support a human reads, after which a chosen Tier-1 initiative becomes
a normal product-spec via the existing `product-design` flow.

### Verification (testability seams)

- The skill's output contract is a single file under `docs/horizons/` + an index
  line (R6's "diff is exactly one doc" is the strongest behavioral check).
- The generate→judge independence is structural (separate `subagent_type`s,
  one-judge-per-vision dispatch) and checkable by reading the skill procedure.
- Frontmatter conformance of the new type is enforced by `check.py` (so a
  malformed `horizon` doc fails the gate).
- Dogfood: a real scout run on *this* repo (it is its own first host) producing a
  plausible two-tier horizon is the end-to-end proof — see the ExecPlan. (Note:
  per the "mid-session agents not dispatchable" constraint, the very first
  dogfood may have to drive the panel via general-purpose subagents carrying the
  persona rubric, since a freshly-authored agent type is not registered until the
  next session.)

## Non-goals

- **No auto-enactment.** The scout never creates specs/plans/tickets or edits the
  charter (R6). Turning a Tier-1 pick into work is a *separate*, human-initiated
  `product-design` run.
- **No periodic/cron cadence and no completion-trigger** in v1 — on-demand only.
  (Wiring it to fire when an initiative completes, or quarterly, is a follow-up.)
- **No Workflow-tool dependency.** The fan-out uses ordinary Task-tool subagent
  dispatches; richer orchestration is not required to ship this.
- **No shared landscape-scan cache** — each generator researches within its stance
  (accepting some redundancy) rather than the skill pre-computing a shared scan.
  Add it only if redundancy proves wasteful.
- **No new charter Initiative** — the scout anchors under the existing Methodology
  initiative; it earns its own initiative only if it grows past one skill.
- **Director/main-session-side by intent** — not a mechanically-enforced
  worker-exclusion (see Design Component 1); per-component vendoring scope is
  tracked tech-debt.

## Acceptance criteria

1. `plugin/skills/scout/SKILL.md`, `plugin/agents/workstream-scout.md`, and
   `plugin/agents/vision-judge.md` exist, pass structure lint (name+description,
   grounding cited), and the two personas declare the tools above.
2. A scout run (dogfooded on this repo) produces a `docs/horizons/` doc with both
   tiers populated: ≥3 distinctly-stanced visions, each with an independent judge
   score sheet and a tier; Tier 2 (if any) names the strained axiom/clause; Tier 1
   items carry a suggested first wedge. External citations are present (R3).
2b. The run's diff is exactly the new horizon doc + its index line — no spec,
   plan, ticket, or charter edit (R6).
3. `type: horizon` is in the KNOWLEDGE_FORMAT type table (self + seed); the new
   doc passes `check.py`.
4. `python3 plugin/scripts/check.py` is GREEN; `lint_base` OK (seed edit synced).

## Open factors — triage & resolution

- **Authority / can it touch the Mission?** *(product fork — escalated.)*
  **Resolved: strategist with a challenge license** — Tier-1 within Mission+axioms,
  Tier-2 foundational challenges escalated, never enacted (R4/R5). Keystone: the
  axiom-screen routes rather than only filters.
- **Structure of "multiple visions evaluated"?** *(escalated.)* **Resolved:
  divergent panel + independent judges + own-context synthesis** (R2/R4, Design
  Component 1).
- **Two personas vs. one?** *(mechanical — decided.)* Keep `vision-judge` separate
  from `workstream-scout`: a generator grading its own moonshot defeats the
  evaluation. Independence > minimalism here.
- **Cadence?** *(mechanical — decided.)* On-demand v1; periodic is a Non-goal.
- **Output home?** *(mechanical — decided.)* New `type: horizon` under
  `docs/horizons/`, distinct from specs/plans (it is pre-spec decision support).
- **Names** (`scout` / `workstream-scout` / `vision-judge` / `horizon`) are
  provisional — cheap to rename; flagged for spec-review.

## Follow-ups (captured, not built)

- Periodic / completion-triggered scout (fires when an initiative lands, or on a
  cadence) — turns on-demand strategy into standing strategy.
- A shared landscape-scan pass (one outward research sweep all generators share),
  possibly via the `deep-research` skill, if per-stance research proves redundant.
- A lightweight loop from a chosen Tier-1 pick into a seeded `product-design` run
  (still human-initiated — just less retyping).
- Sibling of the charter follow-ups: a `nav.py` "Mission-distance" view and an
  axiom-violation lint would give the judge sharper, partly-mechanical inputs.
