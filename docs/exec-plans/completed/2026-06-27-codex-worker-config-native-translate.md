# Codex-worker config — native-translate (Phase 1: skills + agents + trust)

## Goal
A Codex worker dispatched by the Director picks up the harness methodology through **Codex's own native loaders**, exactly as the Claude worker does through `settingSources`. Concretely, after `install_worker_methodology` runs on a workspace clone:
- the methodology + workspace **skills** are discoverable by Codex (present under `<ws>/.agents/skills/`, the only repo path Codex scans) AND still by the Claude worker (`<ws>/.claude/skills/`);
- the review/gardener **agents** are dispatchable by Codex as first-class subagents (present under `<ws>/.codex/agents/<name>.toml` with valid `name`/`description`/`developer_instructions`) AND still by the Claude worker (`<ws>/.claude/agents/<name>.md`);
- the workspace is **trusted** so Codex actually loads its project `.codex/` config layer (without trust, `.codex/agents/*.toml` are silently ignored);
- the **inert** copies that today's code writes but Codex never reads (`<ws>/.codex/skills/`, `<ws>/.codex/agents/*.md`) are gone.

Definition of done (observable): `python -m unittest tests.test_director_run` passes with assertions on the NEW layout; and a live (or mock) Codex worker, launched by the Director in a clone, can spawn a vendored review persona by name at the completion gate. Hooks are explicitly **out of scope** (Phase 2).

## Context
- **Why this exists** — research this session (memory `codex-worker-config-surface`) established that today's `director/run.py install_worker_methodology` copies skills→`<ws>/.codex/skills/` and review agents→`<ws>/.codex/agents/*.md`, but the real Codex CLI reads **neither**: it scans skills only from `.agents/skills/` (repo) + `$HOME/.agents/skills`, and loads subagents only from `.codex/agents/*.toml` (project, **trust-gated**) / `~/.codex/agents/*.toml`. So the Codex worker has been following methodology via prompt + AGENTS.md + reading SKILL.md as plain files, not via native loaders — asymmetric to the Claude worker (`worker-runtime/harness/src/config/resolveOptions.ts:23`, `settingSources` default `["user","project","local"]` reads `.claude/{skills,agents,settings.json}` from disk).
- **Direction decided this session** — option **A (native-translate)**: translate each surface to what Codex's loader consumes, with `.agents/skills` as Codex's skills home and a `.md`→`.toml` translation for agents. (Alternatives B "plugin packaging" and C "prompt/AGENTS.md-driven" were considered and deferred — see Decision log.)
- **Codex facts (verbatim-grounded, developers.openai.com, codex-cli 0.142.0 local / repo pins 0.139.0):**
  - Skills: `Codex scans .agents/skills in every directory from your current working directory up to the repository root.` `.codex/skills/` is **not** a discovery path. `SKILL.md` (name+description frontmatter, the agentskills.io open standard) is the same format the harness already authors.
  - Subagents: `add standalone TOML files under ~/.codex/agents/ … or .codex/agents/ for project-scoped agents`; `Every standalone custom agent file must define: name / description / developer_instructions`; optional `model`/`model_reasoning_effort`/`sandbox_mode`/`mcp_servers`/`skills.config`. `Codex only spawns subagents when you explicitly ask it to.`
  - Trust: project `.codex/` config layers (incl. custom agents) load only when the project is trusted — `Untrusted projects skip project-scoped .codex/ layers, including project-local config, hooks, and rules.` Config key: `projects.<path>.trust_level = "trusted"`.
  - Claude side (confirmed): Claude Code reads `.claude/skills/`, **not** `.agents/skills/` (open issue anthropics/claude-code#31005) — so skills cannot be unified into one shared dir; they are shared *content*, dual-*placed*.
- **Code touched** — `director/run.py` (`install_worker_methodology`, `_VENDORED_SOURCES`, `_WORKER_ROOTS`, `_exclude_injected_methodology`, `_command`); `director/worker/autonomy.py` + `director/config.py` (trust `-c`); `plugin/skills/execplan/SKILL.md` (runtime-neutral persona dispatch); `tests/test_director_run.py`; `docs/SECURITY.md` (trust surface). Authority docs: `ARCHITECTURE.md` (Director is centralized, not ported — so Codex-specific vendoring belongs here, not in the portable seeds), `docs/SECURITY.md` T-rules.

## Approach (self-generated alternatives)
- **A — native-translate (chosen).** Director writes each surface where Codex's loader reads it: skills→`.agents/skills/`, agents→`.codex/agents/*.toml` (translated from the `.md` single source at vendor time), workspace trusted via a launch `-c`. Tradeoff: adds a `.md`→`.toml` translator + a trust knob + a new (small) security surface (trust loads the cloned repo's own `.codex/config.toml`); upside: Codex worker becomes a real native citizen, symmetric to Claude, no drift (single `.md` source).
- **B — plugin packaging.** Ship the methodology as a Codex plugin (`.codex-plugin/plugin.json` bundling skills+agents+MCP) + a repo marketplace, install into `~/.codex/plugins/cache`. Tradeoff: the "blessed" distribution unit and covers non-Director adoption, but heavyweight (marketplace cache indirection, restart-to-refresh, copies out of the workspace) and overkill for the per-ticket-clone model. **Deferred** — a possible 2nd layer on top of A later.
- **C — prompt/AGENTS.md-driven.** Don't use native loaders at all; make AGENTS.md + the ticket prompt point the worker at skill/doc files it reads directly (closest to today's de-facto behavior). Tradeoff: simplest, zero new surface, but loses progressive disclosure + `$skill` ergonomics + first-class subagent dispatch — keeps the asymmetry the Goal exists to remove. **Rejected** as the target.
- **Chosen: A** — it is the minimal change that makes the Codex worker load methodology the way the Claude worker does, and it exploits Codex's deliberate cross-compat with the agentskills.io open standard (so the `SKILL.md` bodies are reused verbatim).
