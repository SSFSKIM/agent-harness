# docs-sync — unified curated-doc maintenance (currency + forgetting)

## Goal
After this, the harness can UPDATE and RETRACT curated docs, not just append.
Observable end state:
- Completing an ExecPlan whose diff makes a doc stale (e.g. a renamed symbol, a
  changed default, a removed feature) → the completion gate's `docs-sync` step
  flags it with evidence (file:line), AUTO-FIXES the mechanical kinds, and REPORTS
  the semantic ones; the machine never auto-edits free prose.
- A dreaming run that drops a session from selection → the docs that session
  authored (located via journal `[routed]` provenance) are retracted/softened.
- `python3 plugin/scripts/check.py` GREEN throughout; every docs-sync edit is a
  git-visible commit.

## Context
Spec design (what/why, the mechanical-4 whitelist, the two scopes, gate
integration): `docs/design-docs/docs-sync.md` — read it first. This EXTENDS the
dreaming write pattern: `plugin/scripts/dream_router.py` already has the read-only-
agent-proposes + deterministic-allowlist-applicator shape and the symlink-safe write
guard (`_within_repo_no_symlink`) this reuses. The completion gate is the
`execplan` skill's review step (where review-arch/reliability/security run); the
provenance source is the `docs/journal/` `[routed] … -> docs/X` lines the router
writes. Follows `docs/PLANS.md`.

## Milestones
- [x] M1 (safety core) Maintenance-plan schema + deterministic risk re-validator +
  the EDIT/DELETE applicator. Build the load-bearing safety FIRST, before any agent.
  `plugin/scripts/docs_sync.py`: a plan item is `{target, kind
  (missing|outdated|retract|structural), evidence, change, risk}`. `apply_plan(root,
  plan, now) -> {applied, report}` RE-VALIDATES each item's risk deterministically
  and auto-applies ONLY the mechanical 4 — (1) regenerate a generator-owned section,
  (2) set a frontmatter field, (3) a verbatim rename swap (old+new given AND old
  found exactly), (4) a `retract` DELETE attributable via journal provenance —
  onto the docs allowlist, through the shared symlink-safe within-repo guard
  (extract `dream_router._within_repo_no_symlink` to `harness_lib`), then `check.py`
  + batch rollback on red. Everything else → the report. Acceptance: stubbed plans →
  each mechanical kind edits correctly; a semantic item (free-prose rewrite) goes to
  the report NOT the docs; a semantic item mislabeled `mechanical` is downgraded; the
  symlink guard refuses a symlinked target; a check.py-red batch rolls back. ~12
  deterministic tests.
- [x] M2 Change-driven scope builder. `build_change_scope(root, base="main") ->
  {changed_files, changed_symbols, removed}` from `git diff --no-color base...HEAD`
  (public surface: exports/config/defaults/CLI/behaviors, with file:line evidence).
  Pure git/text, no agent. Acceptance: a fixture repo with a known diff → the exact
  changed surface; a no-op diff → empty scope. Tests with a temp git repo.
- [ ] M3 (PoC) Audit agent + report. A READ-ONLY agent (`Read,Glob,Grep,LS`, cwd=repo,
  stdin) given the change scope runs openai's doc-first + code-first passes and emits
  the maintenance plan (JSON). New templates `skills/docs-sync/templates/audit_{system,
  input}.md` (the kinds, the risk rubric, evidence-required, DATA guard). `audit(scope,
  spawn=spawn_audit, templates=None) -> plan`. Acceptance (command → output): seed a
  real doc-staleing change (e.g. rename a symbol the docs reference) → run the live
  audit → a plan with an `outdated` item citing the doc + the code evidence → the M1
  applicator mechanical-fixes the rename (or reports if semantic); `check.py` GREEN.
  Deterministic tests stub `spawn_audit`. Technical contract: `spawn_audit(prompt,
  model, cwd, timeout)`.
- [ ] M4 Completion-gate integration. A `docs-sync` skill + `docs_sync.run(root,
  scope)` orchestration wired as a completion-gate REVIEW step (alongside
  review-arch/reliability/security): on ExecPlan completion it audits the diff,
  auto-applies mechanical fixes (committed), and surfaces semantic findings in the
  review. Respects "only check.py blocks a commit" — docs-sync never hard-blocks;
  unaddressed semantic findings are surfaced (v1) for the human/gate. Update the
  `execplan` skill's review checklist + AGENTS.md step 5. Acceptance: completing an
  ExecPlan whose diff staleifies a doc → docs-sync flags + mechanical-fixes/reports;
  a clean diff → no findings.
- [ ] M5 (v1.1, thin) Provenance-forgetting scope. `build_provenance_scope(root,
  dropped_threads) -> targets` from the journal `[routed] … -> docs/X` lines of the
  given dropped sessions; `dream_run` after consolidation calls `docs_sync.run` with
  this scope (retract/soften the unsupported content; DELETE only the attributable
  entries). Same engine, only the scope input changes. Acceptance: a journal with a
  `[routed]` line for a now-dropped thread → the scope names `docs/X` → the audit
  proposes a `retract` → M1 applies it (attributable DELETE) or reports.
- [ ] M6 Completion gate. self-review + codex (gpt-5.5/high per CLAUDE.md) +
  review-security (touches the docs write surface), until SATISFIED. Fill Outcomes;
  move to completed/.

## Decision log
- 2026-06-14: build the deterministic safety core (M1: risk re-validator +
  mechanical-4 + EDIT/DELETE + rollback) BEFORE the audit agent (M3). Why: editing
  curated docs is the risky new capability; the whitelist + guards must be proven in
  isolation so the agent can never be the thing that poisons prose.
- 2026-06-14: extract `_within_repo_no_symlink` to `harness_lib` (shared by the
  dreaming router + docs-sync applicator). Why: both are deterministic doc writers
  that must be symlink-safe; one implementation, one test surface.
- 2026-06-14: v1 = change-driven (M1-M4), v1.1 provenance = a thin M5 (same engine,
  scope swap). Why: change-driven is higher-frequency value and exercises the risky
  EDIT/DELETE path; provenance forgetting is then a small scope add.
