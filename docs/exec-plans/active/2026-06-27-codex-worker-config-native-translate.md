# Codex-worker config — native-translate (Phase 1: skills + agents + trust)

## Goal
A Codex worker dispatched by the Director picks up the harness methodology through **Codex's own native loaders**, exactly as the Claude worker does through `settingSources`. Concretely, after `install_worker_methodology` runs on a workspace clone:
- the methodology + workspace **skills** are discoverable by Codex (present under `<ws>/.agents/skills/`, the only repo path Codex scans) AND still by the Claude worker (`<ws>/.claude/skills/`);
- the review/gardener **agents** are dispatchable by Codex as first-class subagents (present under `<ws>/.codex/agents/<name>.toml` with valid `name`/`description`/`developer_instructions`) AND still by the Claude worker (`<ws>/.claude/agents/<name>.md`);
- the workspace is **trusted** so Codex actually loads its project `.codex/` config layer (without trust, `.codex/agents/*.toml` are silently ignored);
- the **inert** copies that today's code writes but Codex never reads (`<ws>/.codex/skills/`, `<ws>/.codex/agents/*.md`) are gone.

Definition of done (observable): `python -m unittest tests.test_director_run` passes with assertions on the NEW layout; and a live (or mock) Codex worker, launched by the Director in a clone, can spawn a vendored review persona by name at the completion gate. Hooks are explicitly **out of scope** (Phase 2).

## Context
- **Why this exists** — research this session (memory `codex-worker-config-surface`) established that today's `director/run.py install_worker_methodology` copies skills→`<ws>/.codex/skills/` and review agents→`<ws>/.codex/agents/*.md`, but the real Codex CLI reads **neither**: it scans skills only from `.agents/skills/` (repo) + `$HOME/.agents/skills`, and loads subagents only from `.codex/agents/*.toml` (project, **trust-gated**) / `~/.codex/agents/*.toml`. So the Codex worker has been following methodology via prompt + AGENTS.md + reading SKILL.md as plain files, not via native loaders — asymmetric to the Claude worker (`worker-runtime/harness/src/config/resolveOptions.ts:23`, `settingSources` default `["user","project","local"]` reads `.claude/{skills,agents,settings.json}` from disk).
- **Direction decided this session** — option **A (native-translate)**: translate each surface to what Codex's loader consumes, with `.agents/skills` as Codex's skills home and a `.md`→`.toml` translation for agents. (Alternatives B "plugin packaging" and C "prompt/AGENTS.md-driven" were considered and deferred — see Decision log.)
- **Codex facts (verbatim-grounded, developers.openai.com, codex-cli 0.142.0 local / repo pins 0.139.0):**
  - Skills: `Codex scans .agents/skills in every directory from your current working directory up to the repository root.` `.codex/skills/` is **not** a discovery path. `SKILL.md` (name+description frontmatter, the agentskills.io open standard) is the same format the harness already authors.
  - Subagents: `add standalone TOML files under ~/.codex/agents/ … or .codex/agents/ for project-scoped agents`; `Every standalone custom agent file must define: name / description / developer_instructions`; optional `model`/`model_reasoning_effort`/`sandbox_mode`/`mcp_servers`/`skills.config`. `Codex only spawns subagents when you explicitly ask it to.`
  - Trust: project `.codex/` config layers (incl. custom agents) load only when the project is trusted — `Untrusted projects skip project-scoped .codex/ layers, including project-local config, hooks, and rules.` Config key: `projects.<path>.trust_level = "trusted"`.
  - Claude side (confirmed): Claude Code reads `.claude/skills/`, **not** `.agents/skills/` (open issue anthropics/claude-code#31005) — so skills cannot be unified into one shared dir; they are shared *content*, dual-*placed*.
- **Code touched** — `director/run.py` (`install_worker_methodology`, `_VENDORED_SOURCES`, `_WORKER_ROOTS`, `_exclude_injected_methodology`, `_command`); `director/worker/autonomy.py` + `director/config.py` (trust `-c`); `plugin/skills/execplan/SKILL.md` (runtime-neutral persona dispatch); `tests/test_director_run.py`; `docs/SECURITY.md` (trust surface). Authority docs: `ARCHITECTURE.md` (Director is centralized, not ported — so Codex-specific vendoring belongs here, not in the portable seeds), `docs/SECURITY.md` T-rules.

## Approach (self-generated alternatives)
- **A — native-translate (chosen).** Director writes each surface where Codex's loader reads it: skills→`.agents/skills/`, agents→`.codex/agents/*.toml` (translated from the `.md` single source at vendor time), workspace trusted via a launch `-c`. Tradeoff: adds a `.md`→`.toml` translator + a trust knob + a new (small) security surface (trust loads the cloned repo's own `.codex/config.toml`); upside: Codex worker becomes a real native citizen, symmetric to Claude, no drift (single `.md` source).
- **B — plugin packaging.** Ship the methodology as a Codex plugin (`.codex-plugin/plugin.json` bundling skills+agents+MCP) + a repo marketplace, install into `~/.codex/plugins/cache`. Tradeoff: the "blessed" distribution unit and covers non-Director adoption, but heavyweight (marketplace cache indirection, restart-to-refresh, copies out of the workspace) and overkill for the per-ticket-clone model. **Deferred** — a possible 2nd layer on top of A later.
- **C — prompt/AGENTS.md-driven.** Don't use native loaders at all; make AGENTS.md + the ticket prompt point the worker at skill/doc files it reads directly (closest to today's de-facto behavior). Tradeoff: simplest, zero new surface, but loses progressive disclosure + `$skill` ergonomics + first-class subagent dispatch — keeps the asymmetry the Goal exists to remove. **Rejected** as the target.
- **Chosen: A** — it is the minimal change that makes the Codex worker load methodology the way the Claude worker does, and it exploits Codex's deliberate cross-compat with the agentskills.io open standard (so the `SKILL.md` bodies are reused verbatim).

## Assumptions & open questions (self-interrogation)
- Assumption: **skills discovery is NOT trust-gated** (it is a repo directory scan, not a `.codex/` config layer), so `.agents/skills/` loads without trusting the workspace; only `.codex/agents/*.toml` need trust. If wrong, M1 skills also need M3's trust — caught by the M1 behavioral check.
- Assumption: a launch-command `-c projects."<ws_abs>".trust_level="trusted"` is honored by `codex app-server` the same way the existing `-c approvals_reviewer=...` overrides are (`autonomy.py`). If the key shape differs, M3's PoC catches it before the real wiring.
- Assumption: the `tools:` frontmatter on our review personas (`Read, Grep, Glob, Bash`) has **no** direct Codex equivalent (Codex governs tool surface by `sandbox_mode` + `mcp_servers`, not an allowlist); we approximate with `sandbox_mode` derived from tools (Edit/Write present → `workspace-write`, else `read-only`). Recorded as a deliberate lossy mapping.
- Assumption: the Director vendors agents **uniformly** today (all of `plugin/agents/`), so M2 translates the full set rather than curating a worker-only subset — curation would be a separate decision and risks drift. doc-gardener (Edit/Write/Bash) → `workspace-write`; the rest → `read-only`. The two scout personas (`vision-judge`, `workstream-scout`) carry web tools; their `.toml` gets `read-only` and relies on the worker's existing network posture (no per-agent MCP wired in Phase 1).
- Open: should the inert `.codex/skills/` + `.codex/agents/*.md` be *deleted* or just *stopped*? → resolved autonomously: **stop writing them** (per-destination vendoring no longer targets those paths) AND the idempotent target-removal already in `install_worker_methodology` cleans a stale one on the next run; no separate migration needed. Recorded in Decision log.
- Open (Taste — escalate-worthy, but defaulting): the runtime-neutral persona-dispatch wording in `execplan` SKILL.md (M4). Default chosen: keep the existing Claude `subagent_type:` instruction AND add a parallel "on Codex, ask Codex to spawn the `<name>` agent" sentence, gated by a single "whichever your runtime exposes" note (mirrors the wording already in the completion-gate step). If the human prefers a different framing, that is the one genuine taste call — surfaced at the gate, not blocking the build.

## Milestones
- **M1 — Per-destination vendoring; skills dual-placed; inert `.codex/skills` dropped.** Rework `install_worker_methodology` (`director/run.py`) from "copy every source into every `_WORKER_ROOTS` root" to an explicit **destination table**: methodology+workspace skills → `<ws>/.claude/skills/` AND `<ws>/.agents/skills/`; (agents handled in M2). Drop `.codex/skills/` from the targets. Update `_exclude_injected_methodology` so the `.git/info/exclude` patterns cover `/.agents/skills/` and `/.claude/skills/` (and no longer `/.codex/skills/`). Keep the symlink-refusal + idempotent target-removal safety for every new destination. At the end, a vendored workspace has identical skill trees under `.claude/skills/` and `.agents/skills/` and nothing under `.codex/skills/`. Run `python -m unittest tests.test_director_run`; expect the updated placement assertions to pass and an assertion that `.codex/skills/` is absent.
- **M2 — Agent `.md`→`.toml` translation for Codex.** Add a pure translator (new helper in `director/run.py`, unit-tested) that reads a `plugin/agents/<name>.md` (YAML frontmatter `name`/`description`/`tools` + markdown body) and emits a Codex custom-agent TOML: `name` (the frontmatter name), `description`, `developer_instructions` = the markdown body verbatim, and `sandbox_mode` derived from `tools` (Edit/Write → `workspace-write`, else `read-only`). Vendoring writes `<ws>/.codex/agents/<name>.toml` for Codex AND keeps `<ws>/.claude/agents/<name>.md` for Claude; stop writing `.codex/agents/*.md`. Extend the `.git/info/exclude` patterns to `/.codex/agents/` + `/.claude/agents/`. At the end, every vendored persona exists in both formats with a schema-valid TOML. Run `python -m unittest tests.test_director_run`; expect a test that round-trips one persona — parses the emitted TOML, asserts the three required keys are non-empty, `developer_instructions` equals the source body, and `sandbox_mode` matches the tools-derived rule.
- **M3 — Workspace trust so Codex loads `.codex/agents/`.** Thread a trust override into the Codex launch command: extend `director/worker/autonomy.py` `codex_command` (and/or `director/run.py` `_command`) to append `-c projects."<ws_abs>".trust_level="trusted"`, where `<ws_abs>` is the absolute workspace path (the worker's cwd). Validate precedence: the existing `-c` sandbox/approval overrides and thread/start params must still win over anything a cloned repo's own `.codex/config.toml` sets (so trusting the project cannot loosen the worker's posture). PoC first (unknown): a tiny live/mock check that, with trust set, a `.codex/agents/<name>.toml` becomes spawnable; without it, it does not. At the end, the built launch command contains the trust override and a Codex worker in a clone can dispatch a vendored persona. Run `python -m unittest tests.test_director_run` (command-builder assertion) + the PoC transcript captured in Surprises.
- **M4 — Runtime-neutral persona dispatch in the methodology.** Update `plugin/skills/execplan/SKILL.md` (the completion-gate dispatch instructions) so spawning the review personas reads correctly on BOTH runtimes: the Claude `subagent_type:"…review-spec-compliance"` path AND a Codex "ask Codex to spawn the `review-spec-compliance` agent" path, under the existing "whichever your runtime exposes" note. No code; methodology text only — and it is the portable layer (lint S7), so the change is host-agnostic. At the end, a worker on either runtime can execute the completion gate from the SKILL.md alone. Acceptance: re-read the edited section as each runtime would; `python plugin/scripts/check.py` GREEN (doc lints, references).
- **M5 — Security note + gate green.** Record the new surface in `docs/SECURITY.md`: trusting the workspace makes Codex load the **cloned target repo's own** `<ws>/.codex/config.toml` — a new untrusted-input channel — mitigated by M3's precedence guarantee (Director `-c`/thread-start override the worker posture) and the deny-by-default `worker_env`; classify as a new T-rule or an informed-accepted residual with the mitigation cited. Ensure `python plugin/scripts/check.py` is GREEN (numbered-rule renumber if needed — memory `merge-numbered-list-collision-renumber`). At the end the threat model names this surface and the gate is green.

## Decision log
- 2026-06-27: Chose **A (native-translate)** over B (plugin packaging) and C (prompt-driven) — A is the minimal change that restores Claude↔Codex symmetry and reuses the `SKILL.md` open-standard bodies verbatim; B is a deferred 2nd layer, C keeps the asymmetry.
- 2026-06-27: Skills are **dual-placed, not unified** — Claude Code does not read `.agents/skills/` (issue #31005), so `.claude/skills/` (Claude) + `.agents/skills/` (Codex) both get the same content; `.codex/skills/` is dropped (inert).
- 2026-06-27: Agent **single source stays `.md`**; the `.toml` is generated at vendor time inside the Director (the centralized, non-ported layer) so all hosts are covered with zero drift.
- 2026-06-27: `sandbox_mode` for a translated agent is **derived from its `tools`** (Edit/Write → workspace-write, else read-only) — a deliberate lossy map, since Codex has no tool allowlist.
- 2026-06-27: **Hooks deferred to Phase 2** (user decision) — features.hooks + trust + trust-hash friction outweighs current value; skills+agents are the load-bearing methodology delivery.
