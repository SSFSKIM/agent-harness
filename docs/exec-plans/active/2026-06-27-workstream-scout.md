# Build the workstream scout

## Goal

Running the `scout` skill produces a ranked, two-tier `type: horizon` proposal
doc under `docs/horizons/` for the project's next initiatives, generated by a
divergent panel of stance-forced `workstream-scout` generators (with outward web
research) and routed by an independent `vision-judge` (Tier 1 / Tier 2 / drop
against the Mission + Core Axioms), with the skill synthesizing the ranked
result — and it **proposes, never enacts**. **Done is observable when:** the three
new component files exist and pass structure lint; a dogfood run produces a real
`docs/horizons/` doc with ≥3 distinctly-stanced visions, each carrying an
independent judge score sheet + tier, external citations present, Tier-2 items (if
any) naming the strained axiom/clause, and Tier-1 items carrying a first wedge;
the run's diff is exactly that doc + its index (no spec/plan/ticket/charter edit);
`check.py` GREEN and `lint_base` OK.

## Approach (self-generated alternatives)

- **A — components then dogfood (M1 type/scaffold → M2 personas → M3 skill → M4
  run).** Each layer is independently verifiable and the dogfood proves the whole
  flow last. Build order matches dependency (the doc type must exist before a run
  can write one).
- **B — dogfood-first / spike the run with inline prompts, then extract the
  personas+skill from what worked.** Tempting (the prompts are the hard part), but
  it inverts the durable artifact (the files) behind a throwaway run and risks the
  shipped persona prose drifting from what the spike used.
- **Chosen: A.** The files are the deliverable; the dogfood validates them. A also
  lets the always-on reviews see finished components, not spike residue.

Execution mode: **inline** — authoring four tightly-coupled markdown artifacts
where prose consistency (the stance vocabulary, the rubric, the two-tier contract)
must agree across skill ↔ generator ↔ judge; I hold that in context.

  `status: stable` like other authored docs.

  competitor-killer, first-principles-reframe, narrowest-wedge), run in parallel;
  one judge per vision, in parallel; then inline synthesis.

## Milestones

- **M1 — The `horizon` doc type + `docs/horizons/` scaffold.** Register `type:
  horizon` in the type table of `docs/KNOWLEDGE_FORMAT.md` and its seed
  `plugin/skills/harness-init/templates/knowledge-format.md` (edit the seed; `base/`
  re-renders — `lint_base`). Create `docs/horizons/index.md` (a typed index page
  for the proposal artifacts) with conformant frontmatter. At the end the type is
  documented and the output directory exists. Run: `python3 plugin/scripts/check.py`
  and `diff <(render seed) base/...`; expect GREEN + `lint_base: OK`.
- **M2 — The two personas.** Author `plugin/agents/workstream-scout.md` (frontmatter
  `name`, `description`, `tools: Read, Grep, Glob, WebSearch, WebFetch`; body: the
  stance-as-forcing-function mandate, first-principles + outward/analogical research
  with citations, the structured-vision return contract, no self-censoring; grounding
  `docs/CHARTER.md`) and `plugin/agents/vision-judge.md` (frontmatter `tools: Read,
  Grep, Glob, WebFetch`; body: the five-axis rubric, the Tier-1/Tier-2/drop routing
  with the axiom-as-router keystone, independence; grounding `docs/CHARTER.md` +
  `docs/PRINCIPLES.md`). At the end both files exist and pass structure lint. Run:
  `check.py`; expect GREEN (S5 name+description+grounding satisfied).
- **M3 — The `scout` orchestrator skill.** Author `plugin/skills/scout/SKILL.md`
  (frontmatter `name: scout`, `description`): the 5-step procedure (ground in
  charter/roadmap/logs → fan out generators per stance via Task → independent
  per-vision judging → own-context synthesis + ranking → write+commit the horizon
  doc with scoped `git add`), the default stance set, the propose-never-enact fence,
  and the mid-session-agent dogfood note. At the end the skill exists and is
  internally consistent with the two personas (same stance vocabulary, same
  two-tier contract). Run: `check.py`; expect GREEN.
- **M4 — Dogfood: a real horizon for this repo.** Run the scout procedure on
  agent-harness itself: dispatch 4 `general-purpose` generators (carrying the
  workstream-scout rubric, one per stance, with web research) in parallel → dispatch
  4 `general-purpose` judges (carrying the vision-judge rubric, one per vision) in
  parallel → synthesize inline into `docs/horizons/2026-06-27-next-initiatives.md`
  (two tiers, all required fields) → register in `docs/horizons/index.md`. At the
  end a real proposal doc exists proving the end-to-end flow. Run: `check.py` GREEN;
  confirm the doc has ≥3 distinct stances, per-vision score sheets, citations, and
  that this milestone's diff added only the horizon doc + index line (R6).
