# Dreaming v2 — Codex-faithful memory-synthesis pipeline

## Goal
A working, manually-triggered 2-phase memory-synthesis pipeline that replicates
Codex's `codex-memories`. Demonstrable end state: running `/dream` (or
`dream_run.py`) against a set of past Claude session transcripts (a) extracts a
high-signal `raw_memory` per eligible transcript into a `sqlite` store — and
produces NOTHING for a low-signal transcript (the no-op gate) — then (b)
consolidates the selected rows into a self-contained memory workspace, where a
locked-down agent writes `MEMORY.md` + `memory_summary.md` (first line exactly
`v1`), with a second run that adds a transcript surgically updating those files
and a removed transcript surgically forgetting its unsupported content (git-diff
forgetting). The locked-down agent provably cannot write outside the workspace
(post-hoc scope check reverts it). The full spec + the Codex→harness mapping is
`docs/design-docs/dreaming-v2.md`.

## Decision log
- 2026-06-13: workspace at `.claude/harness/memories/` (gitignored runtime +
  nested `.git` baseline) — Codex-faithful (its memory folder is a local git
  repo outside the project), keeps dreaming artifacts out of the lint-governed
  `docs/memory/`. Trade-off: synthesized memory is not repo-tracked/reviewable
  (matches Codex). Reversible to an in-repo location if the user prefers review.
- 2026-06-13: path-scoping the locked-down agent via a post-hoc git scope check
  (revert out-of-workspace writes) — Claude Code has no `writable_roots`
  sandbox flag like Codex; the post-check is the faithful equivalent and also
  closes the open "imprint child unscoped Write" tracker item.
- 2026-06-13: dropped Codex's rate-limit guard (no local backend quota) and the
  multi-worker lease contention (single-dev scale) while keeping the `jobs`
  schema; usage table starts empty (read-path off) → selection falls back to
  recency, exactly Codex's never-used path.
